    数据分布 Data Distribution #

    Bucket 是读写的最小存储单元，每个 bucket 目录包含一个 LSM tree。

    Fixed Bucket #

    配置一个大于 0 的桶数，即使用固定桶模式，通过计算 Math.abs(key_hashcode % numBuckets) 来确定记录所属的桶。

    桶的重新调整只能通过离线流程完成，详见 调整桶数量。

    桶数量过多会导致大量小文件，桶数量过少则写入性能较差，需合理权衡。

    Dynamic Bucket #

    主键表的默认模式，或配置 'bucket' = '-1'。

    先到的数据会落入旧桶，新到的数据会落入新桶，桶与键的分布取决于数据到达的顺序。Paimon 维护索引以确定每个键对应的桶。

    Paimon 会自动扩展桶的数量。

    • 选项1：'dynamic-bucket.target-row-num'：控制每个桶的目标行数。
    • 选项2：'dynamic-bucket.initial-buckets'：控制初始化的桶数量。
    • 选项3：'dynamic-bucket.max-buckets'：控制桶的最大数量。
    动态桶模式仅支持单个写入作业。请勿启动多个作业写入同一分区（否则可能导致数据重复）。即使启用 'write-only' 并启动专门的压缩作业，也无法解决此问题。

    Normal Dynamic Bucket Mode #

    当更新操作不跨分区（无分区，或主键包含所有分区字段）时，动态桶模式使用哈希索引维护键到桶的映射，相比固定桶模式需要更多内存。

    性能表现：

    1. 一般不会有性能损失，但会有额外内存消耗，约 1 亿 条目占用 1 GB 内存，非活跃分区不占用内存。
    2. 对于更新频率较低的表，推荐使用该模式以显著提升性能。

    Normal Dynamic Bucket Mode 支持排序压缩（sort-compact）以加速查询。详见： Sort Compact.

    Cross Partitions Upsert Dynamic Bucket Mode #

    当需要跨分区更新（主键不包含所有分区字段）时，动态桶模式会直接维护键到分区和桶的映射，使用本地磁盘，并在启动流写入作业时通过读取表中所有已有键来初始化索引。不同的合并引擎行为如下：

    1. Deduplicate（去重）：从旧分区删除数据，向新分区插入数据。
    2. PartialUpdate 和 Aggregation（部分更新与聚合）：向旧分区插入新数据。
    3. FirstRow（首行保留）：如果已有旧值，则忽略新数据。

    性能方面：对于大数据量表，性能会明显下降，且索引初始化时间较长。

    如果你的 upsert 不依赖太旧的数据，可以考虑配置索引 TTL 来减少索引数量和初始化时间：

    • 'cross-partition-upsert.index-ttl'：rocksdb 索引和初始化的 TTL，避免维护过多索引导致性能持续恶化。

    但请注意，这样可能会引起数据重复。

    Postpone Bucket #

    延迟桶模式通过配置 'bucket' = '-2' 启用。
    该模式旨在解决难以确定固定桶数的问题，并支持不同分区使用不同的桶数。

    写入表中的所有记录会先存放在各分区的 bucket-postpone 目录中，暂时对读者不可见。

    要将记录移动到正确的桶中并使其可读，需要运行压缩（compaction）作业，详见 compact 流程 procedure。

    首次压缩分区的桶数通过选项 postpone.default-bucket-num 配置，默认值为 4。

    当你觉得某些分区的桶数过小时，也可以运行调整桶数量（rescale）作业，详见 rescale 流程 procedure。

    选择分区字段 Pick Partition Fields #

    在数据仓库中，以下三种字段可以定义为分区字段：

    • 创建时间（推荐）：创建时间通常是不可变的，因此可以放心地将其作为分区字段并加入主键。
    • 事件时间：事件时间是原始表中的字段。对于 CDC 数据，比如从 MySQL CDC 同步的表或 Paimon 生成的变更日志，它们都是完整的 CDC 数据，包括 UPDATE_BEFORE 记录。即使主键中包含分区字段，也能实现唯一性（需配置 'changelog-producer'='input'）。
    • CDC op_ts：不能定义为分区字段，因为无法获知前一条记录的时间戳。因此需要使用跨分区 upsert，资源消耗较大。
