    表模式 Table Mode #

    主键表的文件结构大致如上图所示。表或分区包含多个桶（bucket），每个桶是一个独立的 LSM 树结构，包含多个文件。

    LSM 的写入过程大致如下：Flink 检查点时刷新 L0 文件，并根据需要触发压缩以合并数据。根据写入时的不同处理方式，存在三种模式：

    1. MOR（Merge On Read）：默认模式，只执行小规模压缩，读取时需要合并数据。
    2. COW（Copy On Write）：通过设置 'full-compaction.delta-commits' = '1' 启用全量压缩，同步完成合并操作，写入即完成合并。
    3. MOW（Merge On Write）：通过启用 'deletion-vectors.enabled' = 'true'，写入阶段查询 LSM 生成数据文件的删除向量文件，读取时可直接过滤无效行。

    一般主键表推荐使用 Merge On Write 模式（默认合并引擎为 deduplicate）。

    Merge On Read #

    MOR（Merge On Read）是主键表的默认模式。

    在 MOR 模式下，读取时需要合并所有文件，因为所有文件都是有序的，采用多路合并方式，过程中会对主键进行比较计算。

    这里存在一个明显的问题：单个 LSM 树只能使用单线程读取，导致读的并行度受限。如果桶中的数据量过大，可能导致读取性能下降。
    因此，为了优化读取性能，建议根据查询需求合理分析表，设置每个桶的数据量在 200MB 到 1GB 之间。
    但如果桶过小，会产生大量小文件读写，给文件系统带来压力。

    另外，由于存在合并过程，基于过滤器的跳过数据操作不能作用于非主键列，否则会错误地过滤掉新数据，导致旧数据不正确。

    • 写入性能：非常好。

    Copy On Write # 

    ALTER TABLE orders SET ('full-compaction.delta-commits' = '1');

    将 full-compaction.delta-commits 设置为 1，表示每次写入都会进行全量合并，所有数据都会被合并到最高层级。此时读取时无需合并，读取性能最高。但每次写入都需要全量合并，写放大非常严重。

    • 写入性能：非常差。
    • 读取性能：非常好。

    Merge On Write # 

    ALTER TABLE orders SET ('deletion-vectors.enabled' = 'true');

    得益于 Paimon 的 LSM 结构，它具备通过主键查询的能力。写入时可以生成删除向量文件，标识文件中哪些数据已被删除。这在读取时能直接过滤掉无效行，相当于提前完成了合并，不影响读取性能。

    MOR Read Optimized #

    如果不想使用删除向量模式，又希望 MOR 模式下查询足够快，但只能查询到较旧的数据，还可以：

    1. 在写入数据时配置 'compaction.optimization-interval'。
    2. 从读优化系统表 read-optimized system table查询。
      读取经过优化文件的结果，避免了对相同键记录的合并，从而提升读取性能。
